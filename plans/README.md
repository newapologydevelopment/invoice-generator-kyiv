## Project verification reality (read once)

This project has **no build step and no test framework** — that is intentional (the whole
value prop is "open the file, it works"). So plans verify two ways:

1. **Automated syntax gate** (catches broken JS), runnable from the repo root:
   ```
   node -e "const fs=require('fs');const h=fs.readFileSync('index.html','utf8');const m=h.match(/<script>([\s\S]*?)<\/script>/);if(!m)throw new Error('no script block');new Function(m[1]);console.log('SYNTAX OK')"
   ```
   Expected output: `SYNTAX OK`
2. **`grep` assertions** on `index.html` for the specific change (each plan lists its own).
3. **Manual smoke test**: open `index.html` in a browser, open DevTools console, confirm
   **zero errors**, and check the behavior the plan describes. Each plan gives a checklist.

There is **no git repo**, so plans cannot stamp a commit SHA for drift detection. Instead,
each plan inlines a **Current state** excerpt — before editing, confirm the live code still
matches it; if it doesn't, treat that as a STOP condition.

## Findings considered and rejected

- *Add a test/build harness (vitest/jsdom) to establish a verification baseline*: rejected.
  The project's core constraint is "single file, no install, double-click to run." Adding a
  toolchain would violate that for a ~700-line app. The node syntax gate + grep + manual smoke
  test above is the proportionate verification story. Revisit only if the app grows past a few
  thousand lines or gains multiple files.
- *Escape/XSS hardening pass*: rejected as a finding — `render()` already routes all
  user-supplied strings through `esc()` / `nl2br()`, and the app is single-user client-side with
  no server or third-party data. No action needed.
